# 任务系统（Job & Worker）

---

## 概述

Job（任务）是 OctoManger 中批量执行账号动作的核心机制。

```text
创建 Job → 入 Asynq 队列（Redis） → Worker 拾取 → 逐账号执行 → 写入结果
```

Job 系统由两部分组成：
- **API 服务**：接受创建请求，写入数据库，把任务推入 Redis 队列
- **Worker 进程**：从 Redis 队列取任务，执行业务逻辑，回写结果

---

## Job 字段说明

| 字段 | 说明 |
| --- | --- |
| `type_key` | 目标账号类型，必须是 `generic` 类型 |
| `action_key` | 要执行的动作，如 `REGISTER`，必须在该类型的 `capabilities` 中存在 |
| `params` | 传给 Python 模块的额外参数，JSON 对象 |
| `selector` | 选择要处理的账号范围，见下文 |
| `timeout_minutes` | 任务超时分钟数，0 表示使用系统默认值 |

---

## Job 生命周期

```text
queued（已入队）
    ↓  Worker 拾取
running（执行中）
    ↓  所有账号处理完毕
done（完成）/ failed（失败）
```

也可以在执行中手动取消：

```text
running → canceled（已取消）
```

| 状态 | 说明 |
| --- | --- |
| `queued` | 已写入数据库，任务在 Redis 队列等待 Worker 拾取 |
| `running` | Worker 已开始处理 |
| `done` | 全部账号执行完毕，没有失败 |
| `failed` | 至少有一个账号执行失败 |
| `canceled` | 收到取消请求后提前终止 |

## 账号选择（selector）

创建 Job 时，`selector` 决定处理哪些账号。支持的字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `account_ids` | int[] | 按内部数据库 ID 精确指定 |
| `identifiers` | string[] | 按业务标识精确匹配 |
| `identifier_contains` | string | 业务标识包含此字符串 |
| `tags` | string[] | 包含任一标签的账号 |
| `status` | int | 按状态筛选（`0`=禁用，`1`=启用） |
| `limit` | int | 最多处理多少个账号 |

多个条件之间是 **AND** 关系（同时满足）。`limit` 在所有条件都满足后才限制数量。

**示例：精确指定单个账号（调试用）**

```json
{
  "identifiers": ["demo-user-01"],
  "limit": 1
}
```

**示例：处理所有启用状态的账号，最多 100 个**

```json
{
  "status": 1,
  "limit": 100
}
```

**示例：处理标识符包含 `partner-a` 的所有账号**

```json
{
  "identifier_contains": "partner-a"
}
```

## 查看 Job 结果

### 控制台

打开 **Jobs** 页，点击任意 Job 可以查看：
- Job 基本信息（状态、创建时间、params）
- Run History：每个账号的执行结果（状态、result、error）

### API

```
GET  /api/v1/jobs/                     列表
GET  /api/v1/jobs/{id}                 详情（含 runs）
POST /api/v1/jobs/                     创建
POST /api/v1/jobs/{id}/cancel          取消
```

## Worker 工作机制

Worker 是独立进程（`backend/cmd/worker`），通过 Asynq 从 Redis 消费任务。

### 启动 Worker

```bash
cd backend
go run ./cmd/worker
```

Worker 启动时会（默认）执行数据库迁移，可以关闭：

```bash
DATABASE_AUTO_MIGRATE=false go run ./cmd/worker
```

### 并发控制

Worker 的最大并发数由系统配置 `job.max_concurrency` 控制（默认 5）。可以在控制台 **设置 → 系统配置** 中修改。

### 任务超时

每个 Job 的 `timeout_minutes` 字段控制该任务的最长执行时间，超时后 Asynq 会中止任务并标记失败。全局默认超时由系统配置 `job.default_timeout_minutes` 控制（默认 30 分钟）。

### 取消 Job

向 API 发送取消请求后，Worker 会在**处理下一个账号之前**检查取消标记。已经在执行中的账号不会被中断，但后续账号不会继续执行。

## 系统配置（与任务相关）

以下系统配置项存储在数据库中，可通过控制台 **设置 → 系统配置** 修改：

| 配置键 | 说明 | 默认值 |
| --- | --- | --- |
| `job.default_timeout_minutes` | Job 默认超时分钟数 | `30` |
| `job.max_concurrency` | Worker 最大并发任务数 | `5` |

## 常见问题

**Job 一直是 queued 状态**

Worker 没有启动，或 Worker 与 API 使用了不同的 Redis 实例。检查 `REDIS_ADDR` 配置是否一致。

**Job 执行了但 done 里有 failed 的账号**

查看 Jobs 详情的 Run History，每个账号都有详细的 error 信息。通常是 Python 模块返回了 `{"status": "error", ...}`。

**Job 超时**

增大该 Job 的 `timeout_minutes`，或者增大系统配置 `job.default_timeout_minutes`。也可以拆分任务，用 `selector.limit` 控制每批数量。
