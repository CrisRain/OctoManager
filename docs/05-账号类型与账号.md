# 账号类型与账号

---

## 概念关系

```text
Account Type（类型定义）
  └── 定义"这类账号长什么样、能做什么动作"
        ↓
Account（账号实例）
  └── 某个 Account Type 的具体数据
        ↓
Job（任务）
  └── 对这类账号执行某个动作
```

类型是模板，账号是实例，Job 是执行。

---

## Account Type（账号类型）

### 字段说明

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `key` | string | 类型唯一标识，全小写+下划线，创建后不可修改。如 `demo_shop` |
| `name` | string | 显示名称，如"示例商店" |
| `category` | enum | `generic`（进 OctoModule）或 `email`（走邮箱功能） |
| `schema` | JSON | JSON Schema，定义账号的字段结构和校验规则 |
| `capabilities` | JSON | 定义支持的动作列表 |
| `script_config` | JSON | （可选）指定模块入口文件，留空则使用默认路径 |

### category 的区别

| category | 说明 |
| --- | --- |
| `generic` | 账号通过 OctoModule Python 脚本处理，支持 Job、Trigger、Dry Run |
| `email` | 系统内置的 Outlook 邮箱类型，不走 OctoModule，有专属的邮箱管理页面 |

### schema 格式

`schema` 遵循 JSON Schema（Draft 7）格式，控制台用它自动渲染账号表单。

**最小示例：**

```json
{
  "type": "object",
  "properties": {
    "username": { "type": "string", "title": "用户名" },
    "password": { "type": "string", "title": "密码" }
  },
  "required": ["username", "password"]
}
```

**更完整的示例（带枚举和可选字段）：**

```json
{
  "type": "object",
  "properties": {
    "username":  { "type": "string", "title": "用户名" },
    "password":  { "type": "string", "title": "密码" },
    "region":    { "type": "string", "title": "区域", "enum": ["us", "eu", "ap"] },
    "proxy_url": { "type": "string", "title": "代理地址" }
  },
  "required": ["username", "password", "region"]
}
```

`title` 字段决定表单中显示的标签文字。

### capabilities 格式

```json
{
  "actions": [
    { "key": "REGISTER" },
    { "key": "VERIFY" },
    { "key": "LOCK" }
  ]
}
```

`key` 必须全大写，与 Python 模块中的动作名称一致。Job 和 Dry Run 中选择动作时，列表来源于此。

### script_config 格式

留空时，系统使用默认路径：`<OCTO_MODULE_DIR>/<type_key>/main.py`

自定义入口：

```json
{ "entry": "demo_shop/app/entry.py" }
```

路径相对 `OCTO_MODULE_DIR`，不允许 `../` 或绝对路径。

## Account（账号）

### 字段说明

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `type_key` | string | 关联的 Account Type |
| `identifier` | string | 账号的业务唯一标识，如用户名、邮箱、手机号 |
| `spec` | JSON | 账号具体数据，结构由 Account Type 的 `schema` 定义 |
| `status` | int | `0`=已禁用，`1`=已启用 |
| `tags` | string[] | 标签，用于分组筛选 |

### 创建账号

打开控制台 **Accounts** 页，选择类型后，控制台根据该类型的 `schema` 自动渲染表单。

直接通过侧边栏访问某个类型的账号列表时，URL 格式为 `/accounts/:typeKey`。

### 批量操作

在账号列表中勾选账号，可以进行：
- **批量启用 / 禁用**：修改账号状态
- **批量删除**：不可恢复，操作前会弹窗确认

数量较大时，批量操作会异步执行（进入 Asynq 队列），控制台会提示 Job ID。

## Job 中的账号选择（selector）

创建 Job 时，通过 `selector` 指定要处理的账号范围：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `account_ids` | int[] | 按内部数据库 ID 精确指定 |
| `identifiers` | string[] | 按业务标识精确匹配 |
| `identifier_contains` | string | 业务标识包含此字符串（模糊匹配） |
| `tags` | string[] | 标签筛选 |
| `status` | int | 状态筛选（`0` 或 `1`） |
| `limit` | int | 最多处理多少个账号 |

**开发建议：** 调试阶段用 `identifiers` + `limit: 1` 锁定单个账号，验证没问题后再放开范围。

示例：

```json
{
  "identifiers": ["demo-user-01"],
  "limit": 1
}
```

```json
{
  "identifier_contains": "partner-a",
  "status": 1,
  "limit": 50
}
```
