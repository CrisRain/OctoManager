# 邮箱批量注册 — 内部开发文档

## 概述

批量注册邮件账号功能允许调用方通过一次 API 请求，按照指定的域名、前缀和数量自动生成并持久化一批邮件账号。整个流程是**异步**的：API 返回后，实际的账号创建工作由 Asynq Worker 在后台完成，并通过 Job 记录跟踪执行状态。

---

## 完整执行流程

```
客户端
  │  POST /api/v1/email/accounts/batch-register
  │
  ▼
EmailAccountHandler.BatchRegister          (handler/email_account.go)
  │  绑定并基础校验请求体
  │
  ▼
emailAccountService.BatchRegister          (service/email_account.go)
  │  1. 校验 count / status
  │  2. 在 DB 创建 Job 记录（status=queued）
  │  3. 调用 Producer.EnqueueBatchEmailRegister → Redis/Asynq 队列
  │  返回 {queued:true, job_id, task_id}
  │
  ▼ (异步，Worker 进程)
BatchHandler.ProcessEmailBatchRegister     (task/handler/batch.go)
  │  1. 解析 Payload，基础校验
  │  2. 更新 Job status=running
  │  3. 调用 GoEmailBatchRegistrar.Prepare()   ← 纯 Go，无子进程
  │       按 prefix+index@domain 生成地址序列
  │       为每个地址构造 graph_config（含 tenant 推断）
  │       应用 options.graph_config 覆盖
  │  4. 逐条调用 createEmailAccount() → email_accounts 表
  │  5. 更新 Job status=done/failed/canceled
  │  6. 写入 JobRun 记录
```

## API 端点

```
POST /api/v1/email/accounts/batch-register
X-Api-Key: <key>
Content-Type: application/json
```

### 请求体 (`dto.BatchRegisterEmailRequest`)

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `count` | int | 是 | 要生成的账号数量，范围 1–200 |
| `domain` | string | 是（由 Python 校验） | 邮件域名，例如 `example.com` |
| `prefix` | string | 否 | 邮件地址前缀，默认 `mail` |
| `start_index` | int | 否 | 起始编号，默认 1，最小为 1 |
| `provider` | string | 否 | 邮件服务商标识（`outlook`/`gmail` 等），为空时按域名自动推断 |
| `status` | int16 | 否 | 创建后的账号状态：`0`=待验证（默认），`1`=已验证 |
| `operator` | string | 否 | 操作者标识，用于生成 request_id，默认 `batch-operator` |
| `options` | object | 否 | 传给 Python 的扩展选项，见下方说明 |
| `graph_defaults` | object (JSON) | 否 | 批量覆盖 graph_config 中的字段（仅补充不存在的键） |

**`options` 子字段：**

| 字段 | 类型 | 说明 |
|---|---|---|
| `mailbox` | string | 默认监听的邮箱文件夹，默认 `INBOX` |
| `graph_config` | object | 覆盖 Python 生成的 graph_config（字段级合并，Python 生成值优先） |

### 请求示例

```json
{
  "count": 10,
  "domain": "example.com",
  "prefix": "user",
  "start_index": 1,
  "provider": "custom",
  "status": 0,
  "options": {
    "mailbox": "INBOX"
  },
  "graph_defaults": {
    "client_id": "your-client-id"
  }
}
```

生成的地址序列：`user1@example.com`、`user2@example.com`… `user10@example.com`

### 响应体 (`dto.BatchRegisterEmailResponse`)

API 立即返回，账号尚未创建完毕：

```json
{
  "code": 0,
  "data": {
    "requested": 10,
    "generated": 0,
    "created": 0,
    "failed": 0,
    "provider": "custom",
    "accounts": [],
    "failures": [],
    "queued": true,
    "task_id": "asynq-task-uuid",
    "job_id": 42
  }
}
```

通过 `GET /api/v1/jobs/{job_id}` 轮询 Job 状态，`result` 字段包含最终的 `created`/`failed`/`canceled` 统计。

## GoEmailBatchRegistrar 生成逻辑

地址生成和 graph_config 构造现在完全由 Go 原生实现（[service/email_batch_go_registrar.go](backend/internal/service/email_batch_go_registrar.go)），不再启动 Python 子进程。

### 地址生成

```
address = "{prefix}{startIndex + i}@{domain}"    i ∈ [0, count)
```

### graph_config 默认结构

```json
{
  "auth_method":    "graph_oauth2",
  "username":       "user1@example.com",
  "tenant":         "common",
  "scope":          ["https://graph.microsoft.com/.default"],
  "token_url":      "https://login.microsoftonline.com/common/oauth2/v2.0/token",
  "graph_base_url": "https://graph.microsoft.com/v1.0",
  "mailbox":        "INBOX"
}
```

**tenant 推断规则**（`resolveGraphTenant`）：域名为 `outlook.com`、`hotmail.com`、`live.com`、`msn.com` 时取 `"consumers"`，其余取 `"common"`。

### options.graph_config 覆盖

`options.graph_config` 中的字段会**逐键覆盖**到默认 graph_config 上（options 优先），适合注入 `host`、`port` 等 IMAP 字段。

## Worker 处理细节

处理入口：[task/handler/batch.go:299](backend/internal/task/handler/batch.go#L299)

### `graph_defaults` 合并规则

`graph_defaults` 字段（来自原始请求）在 Worker 层做**补充式合并**（`mergeJSONObjects`）：

```
graph_config 最终值 = graph_defaults 补充 GoRegistrar 生成的 graph_config
```

其中 GoRegistrar 生成的值**优先**（已存在的键不会被 `graph_defaults` 覆盖）。`graph_defaults` 只能用来补充 Go 没有生成的字段（如 `client_id`、`client_secret`）。

### provider 推断（`normalizeEmailProvider`）

若 `provider` 字段为空，Worker 按地址域名自动推断：

| 域名 | provider |
|---|---|
| `gmail.com` | `gmail` |
| `outlook.com`, `hotmail.com`, `live.com` | `outlook` |
| `qq.com` | `qq` |
| `163.com` | `163` |
| 其他 | `custom` |

### 账号去重

`createEmailAccount` 将地址转为小写后写入数据库，`address` 字段有唯一索引。重复地址的创建请求会使该条计入 `failed`，不会导致整个任务失败。

### 取消支持

Worker 在逐条创建每个账号**之前**会检查 Job 的最新状态（`isTrackedJobCanceled`）。通过 `POST /api/v1/jobs/{job_id}` 可发出取消信号，Worker 检测到后立即停止并将 Job 标记为 `canceled`。

## Job 生命周期

| 状态值 | 含义 |
|---|---|
| `0` queued | Job 已创建，任务已入队，等待 Worker 拾取 |
| `1` running | Worker 已开始处理 |
| `2` done | 全部成功（`failed == 0`） |
| `3` failed | 至少一条创建失败 |
| `4` canceled | 收到取消请求后提前终止 |

Job 完成后写入 `job_runs` 表，`result` 字段 JSON 示例：

```json
{
  "requested": 10,
  "generated": 10,
  "created": 9,
  "failed": 1,
  "canceled": false
}
```

## 校验规则汇总

| 层级 | 字段 | 规则 |
|---|---|---|
| Go Binding | `count` | 必填，min=1，max=200 |
| Go Binding | `start_index` | 选填，min=0 |
| Go Binding | `status` | 选填，oneof=0 1 |
| Service | `count` | 必须 > 0 且 <= 200 |
| Service | `status` | 必须 0 或 1 |
| Worker（写 DB） | `address` | 合法邮件地址格式，转小写，唯一 |
| Worker（写 DB） | `graph_config` | 必须是合法 JSON 对象 |
| Worker（写 DB） | `status` | 必须 0 或 1 |

## 扩展 graph_config 生成逻辑

若需要为特定邮件服务商生成不同的 graph_config，有两种方式：

**方式 A — 请求时传入 `options.graph_config`**

在请求体中通过 `options.graph_config` 传入需要**覆盖**到默认值之上的字段（options 优先级高于 Go 默认值）。

**方式 B — 请求时传入 `graph_defaults`**

通过 `graph_defaults` 传入需要**补充**（不覆盖已生成字段）的字段，适合注入 `client_id`、`client_secret` 等认证凭据。

**方式 C — 修改 Go 生成逻辑**

在 [service/email_batch_go_registrar.go](backend/internal/service/email_batch_go_registrar.go) 的 `buildDefaultGraphConfig()` 中按域名添加判断分支，为不同服务商返回不同的配置结构。
