# 架构说明

本文介绍 OctoManger 各组件的职责以及一次完整请求的数据流向。

---

## 整体组成

```text
┌─────────────────────────────────────────────────────┐
│                    React 控制台                      │
│              web/  (Vite + shadcn/ui)                │
└────────────────────────┬────────────────────────────┘
                         │ HTTP（/api/v1/*）
┌────────────────────────▼────────────────────────────┐
│                   Go API 服务                        │
│  backend/cmd/api                                     │
│  ├── Gin 路由层 (internal/router)                    │
│  ├── 鉴权中间件 (internal/middleware)                │
│  ├── Handler 层 (internal/handler)                   │
│  ├── Service 层 (internal/service)                   │
│  └── Repository 层（GORM → PostgreSQL）              │
└────────┬───────────────────────────┬────────────────┘
         │ Asynq 任务入队             │ 直接读写
         │ (Redis)                    │ (PostgreSQL)
┌────────▼────────────────┐   ┌──────▼──────────────┐
│     Asynq Worker        │   │     PostgreSQL       │
│  backend/cmd/worker     │   │  账号、任务、配置等  │

│  └── Python 执行桥      │
│     (internal/worker/   │
│      bridge/python.go)  │
└────────┬────────────────┘
         │ subprocess + stdin/stdout JSON
┌────────▼────────────────┐
│   Python OctoModule     │
│  scripts/python/modules │
│  /<type_key>/main.py    │
└─────────────────────────┘
```

## 各组件职责

### React 控制台（`web/`）

- 展示和操作所有管理功能的 Web 界面
- 所有请求通过 `src/lib/api.ts` 统一发出，自动携带 `X-Api-Key` 请求头
- 路由在 `src/App.tsx` 中定义，使用 React Router v6 懒加载

### Go API 服务（`backend/cmd/api`）

- HTTP 服务，所有业务路由挂在 `/api/v1/*`
- 鉴权中间件检查 `X-Api-Key` 请求头，数据库中没有 Admin Key 时自动放行（引导模式）
- 业务逻辑分三层：**Handler → Service → Repository**，Handler 只做参数绑定和响应格式化，Service 写业务逻辑，Repository 写数据库操作

### Asynq Worker（`backend/cmd/worker`）

- 独立进程，通过 Redis 消费 Asynq 任务队列
- 负责执行所有异步 Job：批量账号操作、邮箱批量注册等
- 对 `generic` 类型账号，通过 Python 执行桥启动子进程执行 OctoModule 脚本

### Python 执行桥（`internal/worker/bridge/python.go`）

- Worker 调用 Python 脚本的唯一出口
- 启动 Python 子进程，通过 stdin 传入 JSON 输入，从 stdout 读取一行 JSON 输出
- 脚本的 stderr 内容被捕获用于错误上报
- 模块目录的 `.venv` 优先于系统 Python

### OctoModule 脚本（`scripts/python/modules/`）

- 每个 `generic` 类型对应一个目录，默认入口为 `<type_key>/main.py`
- 接收 `{action, account, params, context}` 输入，返回 `{status, result}` 或 `{status, error_code, error_message}`
- 依赖完全隔离，每个模块有独立的 `.venv`

## 一次 Job 执行的完整数据流

以通过控制台触发一个 `REGISTER` Job 为例：

```text
1. 用户在控制台点击"创建任务"
   → POST /api/v1/jobs/

2. JobHandler 接收请求，调用 JobService.Create
   → 在 PostgreSQL 写入 Job 记录（status=queued）
   → 调用 Asynq Producer 将任务写入 Redis 队列

3. Worker 进程从 Redis 队列取出任务
   → 更新 Job status=running
   → 查询满足 selector 条件的账号列表

4. 对每个账号，Worker 调用 Python 执行桥
   → 组装 JSON 输入（action、account、params、context）
   → 启动 Python 子进程，写入 stdin
   → 读取 stdout 中的 JSON 输出

5. Python OctoModule 执行业务逻辑
   → 返回 {"status": "success", "result": {...}}

6. Worker 将执行结果写入 job_runs 表
   → 更新 Job status=done/failed

7. 用户在控制台 Jobs 详情页查看结果
```

## 一次 Webhook 触发的数据流

```text
外部系统 POST /webhooks/:slug
  ↓
TriggerHandler 验证 X-Api-Key（Webhook Key）
  或验证 Authorization: Bearer <token>
  ↓
查找对应 Trigger 配置
  ↓
合并 default_params + extra_params
  ↓
async 模式：创建 Job → 入 Asynq 队列 → 立即返回 {queued: true}
sync  模式：创建 Job → 直接执行 → 等待结果 → 返回 {queued: false, output: {...}}
```

## API 响应格式

所有 `/api/v1/*` 接口统一使用以下格式：

**成功：**
```json
{ "code": 0, "message": "success", "data": { ... } }
```

**失败：**
```json
{ "code": 400, "message": "错误描述", "data": null }
```

前端通过 `code !== 0` 判断失败，`message` 字段直接展示给用户。
